# Vue 프론트엔드 API 기반 리팩토링 프롬프트

너는 Vue 프론트엔드 리팩토링을 수행하는 시니어 프론트엔드 엔지니어다.

현재 프로젝트는 `db.json` 또는 json-server에서 데이터를 가져오고 있으며, 대부분의 비즈니스 로직이 Vue 컴포넌트와 프론트엔드 코드 안에 포함되어 있다. 앞으로는 모든 비즈니스 로직을 Spring 백엔드 서버로 이전할 예정이며, 프론트엔드는 API에서 받은 데이터를 화면에 표시하고 사용자 입력을 전달하는 UI 계층만 담당해야 한다.

목표는 기존 프로그램의 화면, 사용자 흐름, 기능 동작을 최대한 동일하게 유지하면서, 프론트엔드 코드를 API 기반 구조로 전면 리팩토링하는 것이다.

---

## 1. 전체 프로젝트 분석

먼저 프로젝트의 모든 파일을 읽고 다음을 파악해라.

- 프로젝트 구조
- 라우팅 구조
- 사용 중인 상태 관리 방식
- 컴포넌트 간 데이터 흐름
- `db.json`, json-server, axios/fetch 등 데이터 호출 방식
- 각 화면에서 사용하는 데이터
- 각 화면에서 수행 중인 비즈니스 로직
- 필터링, 정렬, 집계, 계산, 상태 변경, 권한 판단, 유효성 판단, 파생 데이터 생성 로직
- 현재 UI 동작과 사용자 플로우

분석 후, 실제 수정 전에 다음 내용을 간단히 정리해라.

- 현재 프론트엔드에 존재하는 주요 비즈니스 로직 목록
- 각 로직을 어떤 API 응답 또는 백엔드 책임으로 대체할지
- 프론트엔드에 남겨도 되는 UI 전용 로직과 제거해야 할 비즈니스 로직의 구분
- API 명세서만으로 구현 가능한 부분과 부족한 부분

---

## 2. API 명세서 기준 리팩토링

/api.md 파일 안에 있는 API 명세서를 기준으로 프론트엔드 코드를 리팩토링해라.

중요한 원칙은 다음과 같다.

- 프론트엔드는 API 명세서에 정의된 요청/응답 구조를 신뢰한다.
- 기존 `db.json` 구조에 맞춘 코드를 유지하지 말고, API 명세서의 DTO/응답 구조에 맞게 데이터 접근 방식을 바꿔라.
- 컴포넌트 내부에서 직접 데이터를 조합하거나 계산하지 말고, 가능한 한 API 응답을 그대로 렌더링해라.
- 화면 표시를 위한 최소한의 formatting, loading 처리, error 처리, empty state 처리만 프론트엔드에서 수행해라.
- 도메인 규칙, 상태 판단, 합계/통계 계산, 권한 판단, 복잡한 필터링/정렬/검색 조건 처리, 데이터 변환은 백엔드가 담당한다고 가정해라.
- API 명세서에 없는 데이터가 현재 화면에서 필요하다면 임의로 프론트엔드에서 계산하지 말고, “추가로 필요한 API 또는 응답 필드”로 정리해라.
- 기존 화면과 동일한 UI 결과가 나오도록 하되, API 명세서만으로 불가능한 부분은 명확히 표시해라.

---

## 3. 프론트엔드 구조 개선 방향

리팩토링 시 다음 구조를 우선 적용해라.

- API 호출 코드는 컴포넌트에서 직접 작성하지 말고 별도 모듈로 분리해라.
- 예: `src/api`, `src/services`, `src/lib/http` 등 프로젝트 구조에 맞는 위치를 사용해라.
- axios 또는 기존 프로젝트에서 사용 중인 HTTP 클라이언트가 있다면 그것을 우선 활용해라.
- 중복 API 호출 코드를 제거해라.
- API endpoint, baseURL, 공통 헤더, 에러 처리는 중앙화해라.
- 컴포넌트는 다음 역할만 담당하도록 단순화해라.
  - API 호출 결과 표시
  - loading/error/empty 상태 표시
  - 사용자 이벤트 발생
  - 라우팅 이동
  - UI 전용 상태 관리
- 기존에 컴포넌트 내부에 있던 비즈니스 로직은 제거하거나 API 응답 기반으로 대체해라.
- 재사용 가능한 UI 로직은 composable로 분리하되, composable 안에도 도메인 비즈니스 로직은 남기지 마라.
- TypeScript 프로젝트라면 API 요청/응답 타입을 명확히 정의해라.
- JavaScript 프로젝트라면 JSDoc 또는 명확한 함수명/구조를 통해 데이터 형태를 알 수 있게 정리해라.

## 4. 환경변수 기반 API 전환

환경변수를 통해 API 서버를 전환할 수 있게 만들어라.

개발 또는 mock 모드에서는 json-server를 사용하고, 실제 백엔드 연동 모드에서는 Spring 서버를 사용한다.

권장 방식은 다음과 같다.

- `.env.development` 또는 `.env.local`에서는 json-server 주소를 사용한다.
- `.env.production` 또는 backend 연동 모드에서는 Spring 서버 주소를 사용한다.
- Vite 프로젝트라면 `VITE_API_BASE_URL` 같은 환경변수를 사용해라.
- Vue CLI 프로젝트라면 `VUE_APP_API_BASE_URL` 같은 환경변수를 사용해라.
- 현재 프로젝트의 빌드 도구를 확인한 뒤 그 방식에 맞춰 적용해라.

예시:

```env
VITE_API_BASE_URL=http://localhost:3000
```

```env
VITE_API_BASE_URL=http://localhost:8080
```

환경변수 이름은 프로젝트 관례에 맞춰 정하되, 모든 API 호출이 이 baseURL을 사용하도록 통일해라.

중요한 조건:

- API baseURL을 컴포넌트마다 하드코딩하지 마라.
- json-server와 Spring 서버 전환 시 프론트엔드 코드 수정 없이 환경변수만 바꾸면 동작해야 한다.
- json-server의 endpoint 구조는 가능한 한 API 명세서의 endpoint 구조와 맞춰라.
- json-server만으로 완전히 재현하기 어려운 endpoint가 있다면, 가능한 범위에서 mock 데이터를 구성하고 한계점을 정리해라.

## 5. `db.json` 목 데이터 재구성

현재 `db.json`에 들어 있는 기존 데이터는 API 명세서에 맞지 않으면 제거하고, API 명세서의 응답 구조에 맞는 mock 데이터로 다시 작성해라.

조건은 다음과 같다.

- API 명세서의 각 응답 DTO에 맞는 mock 데이터를 작성해라.
- 기존 화면을 확인할 수 있을 만큼 충분한 데이터를 넣어라.
- 목록/상세/빈 상태/상태값 차이/필터링 결과 등을 확인할 수 있도록 다양한 케이스를 포함해라.
- mock 데이터는 프론트엔드가 기존 주요 화면을 정상 렌더링할 수 있을 정도로 구성해라.
- json-server가 지원하는 REST 구조를 고려해서 작성해라.
- 만약 API 명세서가 json-server 기본 구조와 맞지 않는다면, `routes.json` 또는 별도 mock strategy가 필요한지 판단해서 제안해라.

## 6. 기존 동작 유지

리팩토링 후에도 기존 프로그램의 사용 흐름과 화면 결과가 최대한 동일해야 한다.

다음 항목을 반드시 확인해라.

- 기존 라우트가 정상 동작하는지
- 각 페이지가 정상 렌더링되는지
- 목록 화면이 API 데이터로 표시되는지
- 상세 화면이 API 데이터로 표시되는지
- 등록/수정/삭제 같은 사용자 액션이 있다면 API 호출 방식으로 변경되었는지
- loading 상태가 표시되는지
- API 실패 시 error 상태가 표시되는지
- 데이터가 없을 때 empty 상태가 표시되는지
- 기존 필터, 검색, 정렬 기능이 있다면 API 명세서 기준으로 동작하는지
- API 명세서에 해당 기능이 없다면 필요한 API로 정리했는지

단, 프론트엔드에서 유지하면 안 되는 비즈니스 로직은 억지로 남기지 마라. 기존 기능 재현을 위해 추가 API가 필요하다면 코드에 임시 로직을 넣는 대신, 필요한 API를 문서화해라.

---

현재 화면을 기존과 동일하게 구현하기 위해 API 명세서만으로 부족한 부분이 있다면 아래 형식으로 정리해라.

```markdown
- API 명세서에 부족한 점:
- 제안 API:
- 요청 파라미터:
- 응답 예시:
- 이유:

### 2. 필요한 기능

- 화면:
- 현재 필요한 데이터:
- 기존 프론트엔드에서 처리하던 로직:
- API 명세서에 부족한 점:
- 제안 API:
- 요청 파라미터:
- 응답 예시:

```

예를 들어 다음과 같은 경우는 반드시 추가 API 또는 필드로 제안해라.

- 화면에서 합계, 평균, 통계, 상태값, 진행률을 보여줘야 하는 경우
- 복잡한 검색/필터링/정렬이 필요한 경우
- 프론트엔드에서 여러 데이터를 조합해야만 화면을 만들 수 있는 경우
- 권한 또는 사용자 상태에 따라 버튼 표시 여부가 달라지는 경우
- 상세 화면에 필요한 데이터가 목록 API 응답에 없는 경우
- 기존 화면에 표시되던 필드가 API 응답에 없는 경우
- 기존 UX를 유지하려면 별도 endpoint가 필요한 경우

## 8. 코드 수정 원칙

코드를 수정할 때는 다음 원칙을 지켜라.

- 불필요한 대규모 UI 디자인 변경은 하지 마라.
- 기존 CSS, 레이아웃, 컴포넌트 구조는 가능한 한 유지하되 데이터 흐름과 비즈니스 로직 중심으로 개선해라.
- 사용하지 않는 import, 변수, 함수, mock 로직은 제거해라.
- API 호출 실패를 무시하지 말고 사용자에게 표시 가능한 error 상태를 만들어라.
- async/await 또는 Promise 처리는 일관되게 작성해라.
- 동일한 API 호출 패턴이 반복되면 공통 함수로 분리해라.
- 기존 테스트가 있다면 깨지지 않도록 수정해라.
- 테스트가 없다면 최소한 수동 검증 가능한 실행 방법을 정리해라.
- 린트/빌드 오류가 없도록 작성해라.
- 프로젝트의 기존 코드 스타일과 네이밍 컨벤션을 따른다.

## 9. 검증

리팩토링 완료 후 다음을 수행해라.

- 패키지 매니저를 확인한 뒤 적절한 명령어를 사용해 의존성을 설치하거나 확인해라.
- 개발 서버 실행 명령어를 확인해라.
- json-server 실행 방법을 확인하거나 필요한 script를 추가해라.
- 가능하다면 dev mock 모드에서 프론트엔드가 정상 동작하는지 확인해라.
- build, lint, test 명령어가 있으면 실행해서 오류를 확인해라.
- 오류가 발생하면 원인을 분석하고 수정해라.
- 실행하지 못한 명령이 있다면 이유를 명확히 적어라.

## 11. 절대 하지 말아야 할 것

- API 명세서에 없는 데이터를 프론트엔드에서 임의로 만들어 기존 화면을 억지로 맞추지 마라.
- 컴포넌트 내부에 복잡한 비즈니스 로직을 계속 남기지 마라.
- baseURL을 파일마다 하드코딩하지 마라.
- 기존 UI를 불필요하게 재디자인하지 마라.
- API 명세서와 다른 임의의 DTO 구조를 만들지 마라.
- json-server용 mock 구조에 프론트엔드가 종속되게 만들지 마라.
- Spring 서버로 전환할 때 프론트엔드 코드를 다시 고쳐야 하는 구조로 만들지 마라.
- 빌드 오류나 타입 오류를 남기지 마라.

## 12. 충돌 사항 처리

작업 중 API 명세서와 현재 화면 요구사항이 충돌하면, 임의로 판단해서 기능을 삭제하거나 변경하지 말고 “충돌 사항”으로 정리한 뒤 가장 기존 동작에 가까운 임시 구현과 필요한 백엔드 수정안을 함께 제시해라.

---

## 13. 최우선 기준

작업의 우선순위는 다음과 같다.

1. 기존 사용자 경험과 화면 동작 유지
2. API 명세서 기반 데이터 구조로 전환
3. 프론트엔드 비즈니스 로직 제거
4. 환경변수 기반 json-server/Spring 서버 전환
5. 유지보수 가능한 API 호출 구조 정리
6. 부족한 API 또는 응답 필드 명확화

프론트엔드에서 기존 기능을 억지로 재현하기 위해 비즈니스 로직을 새로 만들지 마라. 기존 동작을 유지하기 위해 백엔드 지원이 필요한 경우, 필요한 API와 응답 필드를 명확히 제안해라.
